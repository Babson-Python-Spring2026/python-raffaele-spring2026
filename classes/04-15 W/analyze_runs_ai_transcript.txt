AGENT: Three places people usually get this wrong:

  1. Equal adjacent values. Is [5, 5] a run of length 1 or 2? The fact
     that the definition says "strictly increasing / strictly decreasing"
     strongly implies that equals are NOT part of any monotone run, so
     they should BREAK the current run. You should make that rule
     explicit in your invariants.

  2. Counting runs vs. tracking the longest one. These are two different
     pieces of state. The longest-run length can be updated every time
     you extend the current run. But num_increasing_runs should go up
     exactly once per *maximal* run, at the moment you first recognize
     that a new run has started, not once per element inside it.

  3. longest_run_values. Most implementations track only the length and
     then have no way to reproduce the actual values at the end. You
     need to also store the start index of the best run so you can slice
     it out of nums later.

AGENT: Here are four invariants, ordered from most fundamental to most
subtle. You should state these in your docstring before the code:

  I1. run_len >= 1 at any point after the scan has started. A run
      with zero elements is a contradiction.

  I2. best_inc and best_dec never decrease. They only move up, at the
      exact moment a newly extended current run beats the old best.

  I3. num_inc counts each maximal strictly increasing run exactly once.
      Enforced by setting counted_inc at the first increasing
      transition and clearing it whenever direction changes away from
      'inc' (including on equal). num_dec is symmetric.

  I4. longest_run_values is always in sync with its recorded length.
      That is, when you finally slice nums[start : start + length],
      the length must equal best_inc or best_dec (whichever is the
      overall winner). Enforced by updating start and length together
      in the same code block, never in separate blocks.
